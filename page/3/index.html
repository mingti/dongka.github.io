二进制

Posted on 2024-01-03 Edited on 2025-12-31

补码1

我们首先应该知道进制只是表示数量的一种方式, 无论使用什么进制来计算, 5个苹果吃掉2个苹果的答案一定是3个苹果.

在引入了负数的概念后, 加减法可以互相转化, A-B 等价于 A+(-B), 这仍然不受进制的影响, 5个苹果吃掉2个答案仍然是3个苹果.

先来考虑两位十进制的一个问题:

57-25=32

如果不考虑进位, 即答案只有两个有效位的情况下:

57-25=32 等价于 57+75=(1)32

我们可以加到答案溢出, 并且溢出之后的有效位等于答案. 这样我们就可以使用加法来处理减法.

观察上图,可以得到需要加的值75等于100-25

那为什么是用100来减呢?
因为正确答案32 和溢出答案132差值是100, 需要溢出答案永远比正确答案多一位1. 这多出来的1位则相当于两位有效位最多能表示的个数, 即是100.

这里我们将75称为-25的补码形式, 即为了补到溢出答案相同所需要的码.

基于以上的想法,我们再来看看两个负数相加的情况:

-25+(-63) = -88

换成补码:

75+37 = (1)12

-88补码为12 不考虑进位 112=12,
可以看到仍然是成立的

所以只要把负数换成补码的形式, 我们就能只使用加法来计算正负数之间的加减法.

接下来我们换成二进制, 进制不会影响计算结果. 所以二进制下 A -B = A+(-B)仍然成立,

-B的补码计算仍然= 有效位能表示的最多个数-B

如有8个二进制有效位:

57-25=32

换成补码

(0011 1001) + ( 1110 0111 ) = (0010 0000)

57+231=288

288( 1 0010 0000 )=32( 0010 0000 ) 因为溢出后有效位相等.

那 231 怎么来的呢? 根据我们十进制的算法可以推出:

231= 8个二进制有效位能表示的最多个数(256) - 25

对二进制A取反 = 有效位全部置1 - A

有效位全部置1 + 1 = 有效位能表示的最多个数

所以A补码 = 有效位能表示的最多个数 -A
= 有效位全部置1 +1 - A
=有效位全部置1 - A +1
= 取反 +1

缓存函数的结果

functools.cache会对函数结果做一个缓存,相同的参数调用,直接返回已缓存的值,不会运行函数

functools.lru_cache 运用了最久未使用替换算法, 所以可以设定缓存大小, 如果设置为None,LRU 特性将被禁用且缓存可无限增长, 等价与cache.

能有Pycharm支持是最好的, 不然想到用Pycharm新建一个项目前,还要切出去创建一个虚拟环境, 我的心里就像吃到夹生饭一样难受.

支持下载多个版本的Python也是好的, 但我可能有点偏激了, 现在大部分项目已经使用python3, 支持多个版本可能已经是一个伪需求.

Pycharm支持并且能下载多版本Python. 只有conda, 可它无法提供一个干净的虚拟环境. 总会带一些包.

退而求其次, 使用pyenv管理python, 切换全局python之后, 使用其他工具创建虚拟环境行不行呢.但 而是pyenv通过在PATH前面插入pyenv自己的python路径,从而拦截python命令,并不会更改/usr/bin/python指向的版本. 所以 pycharm 也检测不到.

但我们在基础解释器选择上pyenv的python路径就可以了. 也没必要选shims路径的下python, 这样还得在选之前 使用 pyenv 更改全局python版本, 直接选versions路径下的python也可以.

同理这种方法使用conda也可以, 基础解释器选上conda路径的python, 不勾选继承全局包, 这样就把conda自带的包去掉了.

除了pyenv,conda还有一个rye也可以下载多版本python, 但它不能直接下载一个指定版本的python,需要配合项目路径下其他配置文件来共同管理虚拟环境和包, 如pyproject.toml, .python-version, 其实rye还有很完美的, 但缺少pycharm支持,和pip兼容.

综上所诉,我们找到了一种可以下载管理多版本虚拟环境,并且可以有pycharm支持的方法.就是pyenv+virtualenv
